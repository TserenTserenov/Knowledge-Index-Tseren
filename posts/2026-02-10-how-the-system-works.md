# Как устроена моя система управления знаниями (и почему именно так)

---

## Проблема

У меня 15+ репозиториев. Бот, цифровой двойник, три пака знаний, курсы, стратегия, экосистема. Каждый проект — свой домен, свои термины, свои решения.

Но знания пересекаются. Метод оценки компетенций нужен и боту, и курсу, и ЦД. Архитектура FSM нужна и боту, и платформе. Принцип bounded context используется везде.

Без системы это превращается в хаос:
- Одно и то же описано в 5 местах, везде чуть по-разному
- При изменении обновляешь одно место — остальные устаревают
- Новый проект начинаешь с нуля, хотя знание уже есть
- ИИ-ассистент каждую сессию начинает без контекста

---

## Решение: 4 уровня

```
1. FPF  — универсальные принципы мышления (мета-онтология)
    ↓
2. SPF  — правила формы и процесса (как описывать)
    ↓
3. Pack — знания предметной области (source-of-truth)
    ↓
4. Downstream — код, курсы, агенты, планы (производные)
```

**Ключевое правило: Pack — единственный source-of-truth. Всё остальное производно.**

Бот берёт знания из Pack. Курс берёт знания из Pack. Агент берёт знания из Pack. Если что-то изменилось в домене — меняешь Pack, downstream подтягивается.

---

## Как это выглядит на практике

### Паки (ядро знаний)

Три пака — три области:

| Pack | Область | Что внутри |
|------|---------|-----------|
| PACK-personal | Созидатель | Характеристики, состояния, программы развития |
| PACK-ecosystem | Экосистема | Компоненты, бесшовность, сообщество |
| PACK-digital-platform | ИТ-платформа | Архитектура, агенты, ЦД, протоколы |

Каждый пак — bounded context. Внутри: глоссарий (термины), различения (что с чем нельзя путать), методы, failure modes (типовые ошибки). Термин определён строго внутри границ контекста.

### Downstream (производные)

| Подтип | Что делает | Примеры |
|--------|-----------|---------|
| instrument | Код, боты, сервисы | aist_bot, digital-twin-mcp |
| governance | Планы, реестры, процессы | ecosystem-development, my-strategy |
| surface | Курсы, гайды | docs |

Downstream **не придумывает знания** — берёт из Pack и применяет к конкретной задаче.

### Экзокортекс (память между сессиями)

```
~/.claude/projects/.../memory/
├── MEMORY.md          ← всегда в контексте (план недели, навигация)
├── hard-distinctions.md
├── fpf-reference.md
├── repo-type-rules.md
└── ...                ← подгружаются по необходимости
```

MEMORY.md — компактный (200 строк), всегда загружен. Содержит таблицу РП на неделю, навигацию по файлам памяти, критические правила. Детали — в отдельных файлах, которые Claude читает когда нужно.

### Агент Стратег (автоматизация)

Стратег — ИИ-агент, который работает по расписанию:

| Время | Что делает |
|-------|-----------|
| Пн 00:00 | Собирает коммиты за неделю → WeekReport (для клуба) |
| Пн 4:00 | Читает WeekReport → черновик плана недели |
| Вт-Вс 4:00 | Собирает вчерашние коммиты → план дня |
| По запросу | Закрытие дня, добавление РП, проверка плана |

Стратег не импровизирует — работает по формализованным сценариям из Pack.

### Протоколы сессии (Open → Work → Close)

Каждая сессия Claude Code:
1. **Open** — проверяет: задача совпадает с РП недели? Нет → СТОП, сначала запиши РП
2. **Work** — выполняет задачу + на рубежах проверяет: есть ли знание для записи в Pack?
3. **Close** — собирает captures, обновляет статусы, делает backup экзокортекса

---

## Почему именно так (мои аргументы)

### 1. Знание отделено от реализации

В типичном проекте знания живут в коде, в головах, в Notion, в Slack. Когда меняется реализация или уходит человек — знание теряется.

Pack хранит знание отдельно от кода. Бот можно переписать с нуля — знания останутся. Курс можно переверстать — знания останутся. Формулировка «как устроен домен» не зависит от того, на каком языке написан бот.

### 2. Переиспользование через много проектов

Когда у тебя 15 репо, повторное использование — не роскошь, а необходимость. Без единого источника:
- Бот использует свой список компетенций, курс — другой, ЦД — третий
- При обновлении модели нужно помнить все 15 мест

С Pack: обновил один раз — все downstream обновляются.

### 3. ИИ-агент может работать автономно

Формализованное знание — это не заметки «для себя». Это структура, которую ИИ-агент может читать, интерпретировать и применять. Стратег читает WORKPLAN.md из каждого репо, потому что формат единый. Claude Code проверяет РП в MEMORY.md, потому что таблица структурирована.

Без формализации ИИ каждый раз тратит контекст на «разобраться, что тут происходит». С Pack — сразу работает.

### 4. Capture-to-Pack не даёт знаниям теряться

Самая частая потеря: в процессе работы обнаружил паттерн, failure mode, различение — но не записал. Через неделю забыл.

Протокол Work обязывает проверять на каждом рубеже: есть ли знание для записи? Это не «если заметил» — это обязательная проверка. Мелкое записывается сразу, крупное — при Close.

### 5. WP Gate защищает от бессмысленной работы

Без правила «нет РП — не начинай» легко потратить 3 часа на задачу, которая не в плане, не в приоритетах, не ведёт к результату. WP Gate — шлюз: каждая работа привязана к конкретному рабочему продукту с бюджетом и дедлайном.

### 6. Иерархия масштабируется

FPF → SPF → Pack → Downstream — это не только для 15 репо. Добавить новый домен = создать новый Pack. Паттерны из FPF/SPF уже готовы. Формат из SPF уже готов. Нужно только заполнить содержание.

При 50 репо система работает так же. При 100 — так же. Потому что каждый Pack — автономный bounded context со своим глоссарием и границами.

---

## Чего это стоит

Честно:
- **Overhead на формализацию.** Записать знание в Pack — это работа. Frontmatter, различения, failure modes. Не каждый инсайт стоит формализации
- **Discipline.** Протоколы работают, только если их соблюдаешь. WP Gate бесполезен, если его обходить. Close бесполезен, если его пропускать
- **Начальные инвестиции.** Настроить экзокортекс, написать CLAUDE.md, описать процессы — это не один вечер

Но альтернатива — хаос из 15 репо, где знания дублируются, теряются и противоречат друг другу. Я это проходил. Система окупается.

---

## Итого

Не пытаюсь сделать «идеальную методологию». Решаю конкретную проблему: много проектов, много пересекающихся знаний, ограниченное время, ИИ-ассистент как основной инструмент.

Ответ: формализуй знания один раз (Pack), используй везде (Downstream), не теряй по дороге (Capture-to-Pack), автоматизируй рутину (Стратег), защищай фокус (WP Gate).

Работает? Да. Совершенно? Нет. Лучше, чем без системы? Определённо.
